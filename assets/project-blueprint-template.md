## Project Evaluation Dimensions

Define the scorecard before creating executable tasks. These dimensions become the project review standard and must be reused in `tasks/progress-board.md`.

Scores are not emotional feedback. They represent demonstrated capability/maturity against the user's intended outcome and a reasonable market/productized standard.

| Dimension | Capability Standard | Why It Matters | Initial Score | Target Score | Evidence / How To Judge | Human Review Needed? |
|-----------|---------------------|----------------|---------------|--------------|--------------------------|----------------------|

| | | | / 100 | / 100 | | yes/no |
| | | | / 100 | / 100 | | yes/no |
| | | | / 100 | / 100 | | yes/no |
| | | | / 100 | / 100 | | yes/no |

### Dimension Guardrails
- Define dimensions dynamically for this project. Do not reuse a fixed universal scorecard unless it genuinely fits.
- Keep 5-9 dimensions; too many dimensions dilute review quality.
- Each dimension must be observable through tests, user review, artifacts, or measurable behavior.
- A score means "how much of this capability exists and how far it is from the expected standard," not "how much we like it."
- Mark which dimensions need human review because AI cannot fully judge business taste, market fit, trust, or workflow usefulness.
- Do not add tasks only because they are interesting; every task should improve or unblock at least one dimension.
- Each executable task should primarily improve 1-3 dimensions. If it claims more, split it.
- If the project changes enough to require a new dimension, update this blueprint or record the change in the progress board before dispatching related tasks.
